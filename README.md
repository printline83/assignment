# 상품 예약 및 토스 페이먼츠 결제 시스템

이 프로젝트는 한정된 재고를 다루는 상품의 예약 및 결제 시스템을 구현한 CodeIgniter 3 기반 어플리케이션입니다.
동시성 처리(비관적 락) 및 토스 페이먼츠 연동을 통해 안정적인 커머스 흐름을 제공합니다.

## 핵심 구현 로직 요약

1. **토스 페이먼츠(Toss Payments) 결제 플로우 완벽 연동**
   - 클라이언트 측 JS SDK 호출 및 서버 투 서버(S2S)의 안전한 시크릿 키 결제 검증 구현
   - GET 파라미터로 주소창에 결제 중요 정보가 노출되는 것을 방지하기 위해 Session 일회성 플래시 데이터를 활용해 깔끔한 결제 결과창으로 리다이렉트

2. **재고 관리를 위한 예약 기반 선차감(Pre-Deduction)**
   - 예약(결제창 진입) 시점에 트랜잭션 및 비관적 락(`SELECT ... FOR UPDATE`)을 발동시켜 동시 접속자의 오버부킹(초과 판매)을 원천 차단
   - 한 사용자가 독점 결제를 할 수 있도록 10분의 타이머(결제 유효 시간)를 부여

3. **교착상태 없는 재고 복구(Garbage Collection/Sweeping) 시스템**
   - **Active Sweeping**: 사용자가 능동적으로 '결제 포기'를 누르거나, 토스에서 오류(`failUrl`)가 나거나, 만료시간이 지났는데 결제 버튼을 누를 경우 즉시 재고 환원 및 결제 차단
   - **Lazy Sweeping**: 예약(장바구니) 담기 시도 시, 이전에 타인이 걸어두고 10분이 지난 미결제 허수 데이터를 서브쿼리 레벨에서 감지하고 정리해 재고 자동 부활
   - **Cron 처리**: 백그라운드용 `Cron.php`를 구축하여 독립적인 Row-Level 마이크로 트랜잭션 기반 하드 청소를 통해 메인 결제 스레드와 데드락(Deadlock)이 없도록 설계됨

4. **유연한 취소 및 환불 시스템 (Code Reuse)**
   - **결제 전 예약 취소(포기)**: 사용자가 결제창에서 이탈하거나 능동적으로 취소를 누를 경우, 즉시 상품 재고를 복구하고 예약 상태를 원복합니다.
   - **결제 후 전액 환불**: 결제가 완료된 주문이라도, 관리자나 사용자의 요청에 의해 토스 PG 승인 취소와 동시에 DB 내역을 철회하고 재고를 다시 시장에 공급합니다.
   - **로직 재사용(DRY)**: 위 두 가지 서로 다른 시점의 취소 로직을 하나의 공통 DB 처리 함수로 모듈화하여, 상태 변경과 재고 복구의 정합성을 일관되게 보장합니다.

## 데이터베이스 스키마 구조 (Database Schema)

본 시스템은 재고 무결성과 예약-결제의 유연성을 보장하기 위해 3개의 핵심 비즈니스 테이블을 중심으로 견고하게 연결되어 있습니다.

### 1. `t_Products` (상품 정보)

상품의 고유 정보와 **잔여 재고**를 관리하는 마스터 테이블입니다.

- `f_ProductId` (PK): 고유 상품 번호
- `f_Stock`: 실시간 구매 가능한 잔여 재고 숫자 (예약 시 발생되는 비관적 락(Lock)의 핵심 기준점)

### 2. `t_Reservations` (예약 및 주문 원장)

고객이 '주문하기(결제창 진입)' 버튼을 누르는 즉시 생성되어, 타인의 간섭 없이 단독 결제를 보장해주는 원장 테이블입니다.

- `f_ReservationId` (PK): 시스템에서 발급하는 고유 주문/예약 번호
- `f_Status`: 예약의 라이프사이클 관리. `ED`(결제 대기중), `CF`(결제 완료), `CX`(변심 취소), `EX`(시간 만료)
- `f_ExpiredAt`: 예약 발동 시점으로부터 정확히 +10분이 부여되는 타임아웃 시각

### 3. `t_Payments` (결제 및 승인 이력)

토스 페이먼츠(PG) 서버로부터 정상적으로 거래 대금이 승인(Confirm)된 실제 매출건명세 테이블입니다.

- `f_PaymentId` (PK): 내부 결제 이력 고유 인덱스
- `f_TransactionId` (UNIQUE): 토스의 `paymentKey`가 저장됩니다. 무결성(`UNIQUE KEY`) 제약조건이 걸려있으므로, **웹 브라우저 오류나 악의적인 새로고침으로 인해 동일한 결제 처리가 두 번 이상 중복 매입(Double-charge)되는 것을 완벽히 방탄(Idempotency) 방어**합니다.
- `f_Status`: `SS`(성공), `CX`(결제 전액 환불), `FL`(실패)

> [!TIP]
> **재고 관리 데이터 플로우**
>
> 1. 예약 시점 (`t_Reservations` 생성 & `t_Products` 재고 -1) ➡️ 2. 결제 완료 (`t_Payments` 승인 기록 & 상태값 완료 변경) ➡️ 3. 환불/만료 시 (`t_Payments/Reservations` 취소 변경 & `t_Products` 재고 +1 환원)

## 필수 환경 및 요구사항

- **PHP 버전**: PHP 7.4.33
- **웹 서버 / DB**: Apache 또는 Nginx / MySQL 5.7+ 또는 MariaDB 10.3+

## 설치 방법 및 실행 절차

1. **Git Clone** 후 저장소의 원본(Root) 폴더로 진입합니다.
2. 프로젝트 루트 위치에 `.env` 파일을 새로 생성하고 아래의 데이터베이스 환경 변수를 자신의 로컬 또는 운영 서버 환경에 맞게 기입하세요.

   ```env
   DB_HOST =
   DB_USER =
   DB_PASS =
   DB_NAME =
   ```

3. 동봉된 **`schema.sql`** 파일을 사용하여 데이터베이스의 4개 필수 테이블 구조를 생성(Import)해 줍니다.
4. 결제 유효시간(10분)이 초과된 데이터의 자동 청소를 위해 서버 크론탭(Crontab)에 아래 명령어를 등록하거나 주기적으로 실행합니다. (권장 주기: 1~5분)
   ```bash
   php index.php cron sweep_expired
   ```
